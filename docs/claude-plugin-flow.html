1. 핵심 요약

실행 코드는 “skill 지침”이다

이 plugin은 별도 MCP 서버를 등록하지 않습니다. .mcp.json의 mcpServers는 비어 있고, 핵심 동작은 skills/cmux-agent-harness-loop/SKILL.md의 절차 지침, 계약문서, 템플릿으로 정의됩니다.

Claude가 orchestrator다

Claude/OMC가 사용자 요청을 intake하고 계획을 세우며 실제 파일을 수정합니다. Reviewer인 OMX/Codex는 별도 cmux pane에서 검토 결과를 내지만, 최종 판단과 source 변경 권한은 orchestrator에게 있습니다.

review는 three-channel이다

모든 review는 cmux send로 prompt를 보내고, cmux read-screen으로 pane/sentinel을 관찰하며, docs/changes/REVIEW_*.md.done marker가 생긴 뒤 verdict 파일을 파싱합니다.

한 줄로 말하면

“Claude가 작업을 수행하고, cmux 안의 지속형 Codex/OMX reviewer에게 안전한 방식으로 검토를 맡긴 뒤, GO/STOP/NEEDS_CHECK 결과를 파일 기반으로 확정해 다음 루프 여부를 결정하는 하니스 루프”입니다.

7. Loop Flow: 5개 관찰 가능 상태

이 plugin의 핵심 cursor는 .agent-harness/state.json.stage입니다. 모든 전환 후 stage를 저장해야 하므로, 세션이 끊겨도 이어서 복구할 수 있습니다.

INTAKE

요청 수집 + 계획

요청을 verbatim 저장하고, type을 분류하고, branch/status/diff/docs를 요약해 TASK_SPEC.md를 작성합니다.

EXECUTE

Orchestrator 실행

v0.1에서는 Claude/OMC orchestrator가 직접 source file을 수정합니다. worker pane은 deferred입니다.

REVIEW

Both-channels 검토

reviewer pane을 확보하고 context 9종을 주입한 뒤 verdict file을 기다립니다.

DECIDE

Finding 판정

Claude가 reviewer finding마다 ACCEPT/REJECT/NEEDS_CHECK/DEFER를 결정합니다.

HANDOFF

기록 + 종료 판단

HANDOFF.md, TODO.md, DECISIONS.md를 갱신하고 GO/STOP을 notify합니다.

종료 조건결과이유
last_decision == GO 그리고 remaining_criteria == 0DONE수렴 완료
loop_count >= max_loopsNEEDS_CHECK무한 루프 방지. 기본 max_loops=5

8. Review Flow 상세: send + read-screen + file

가장 중요한 규칙

read-screen에서 sentinel을 보더라도 바로 verdict를 확정하지 않습니다. 진짜 parse trigger는 docs/changes/REVIEW_<loop_id>_<attempt>.md.done marker 존재입니다.

R1

Pane 확보

panes.json.surfaces.reviewer를 읽고, live면 재사용합니다.

R2

죽은 pane 복구

read-screen --lines 5 실패 시 respawn-pane 또는 new-split.

R3

Prompt 전달

이미 실행 중인 interactive omx/codex pane에 cmux send 또는 buffer/paste로 prompt를 주입합니다.

R4

Screen 관찰

cmux read-screen --scrollback --lines 200으로 liveness와 sentinel을 봅니다.

R5

Verdict file

.done marker가 생긴 뒤 REVIEW_*.md를 파싱합니다.

R6

DECIDE

GO/STOP/NEEDS_CHECK를 바탕으로 TODO, decisions, handoff를 기록합니다.

Reviewer에게 주입하는 9개 context

1–3

사용자 원문 요청, classified type, 현재 branch

4–5

git status -s, git diff --stat 및 staged diff stat

6–7

열린 TODO와 최근 decisions digest, 최근 review log 요약

8

현재 loop stage와 loop_id

9

AGENTS.md, CLAUDE.md, README.md, docs/changes/*.md 요약

Redaction

*_API_KEY, *_TOKEN, *_SECRET, *_PASSWORD는 기록 전 redaction합니다.

Verdict write ordering

1. docs/changes/REVIEW_<loop_id>_<attempt>.md.tmp 작성 + flush
2. tmp 파일을 REVIEW_<loop_id>_<attempt>.md 로 atomic rename
3. rename 이후 REVIEW_<loop_id>_<attempt>.md.done marker 생성
4. .done 이후 screen sentinel 출력:
   <<<HARNESS_VERDICT_DONE id=<loop_id>_<attempt>>>>
상황처리
.done timeoutpane respawn/new-split 후 attempt 증가, 재전송. 2회 timeout이면 NEEDS_CHECK + notify
verdict unparseable1회 retry 후에도 single decision token이 없으면 NEEDS_CHECK
질문이 있음NEEDS_CHECK로 표면화. GO/STOP을 임의로 만들지 않음
reviewer pane 사망respawn 실패 시 new-split, attempt bump, idempotent re-send, REVIEW_LOG 기록

9. 대상 프로젝트에 남는 파일과 데이터 흐름

.agent-harness/: runtime control plane

  • harness.yaml: role, transport, max loop, memory path, safety config
  • state.json: 현재 stage, loop_id, attempt, loop_count, convergence, last decision
  • panes.json: orchestrator/reviewer/worker/test/browser surface refs
  • prompts/*.md: orchestrator, reviewer, worker, handoff prompt templates

docs/changes/: shared memory

  • TASK_SPEC.md: 사용자 요청, type, plan, acceptance criteria
  • REVIEW_*.md + .done: reviewer verdict
  • REVIEW_LOG.md: review history
  • DECISIONS.md: orchestrator의 finding 판정과 rationale
  • TODO.md: ACCEPT된 follow-up item
  • HANDOFF.md: 다음 session/loop용 요약
Input

User request

요청 원문이 TASK_SPEC와 reviewer prompt context로 들어갑니다.

Runtime

state + panes

stage cursor와 pane surface가 루프 재개와 reviewer 재사용을 가능하게 합니다.

Review

verdict file

화면이 아니라 파일+done marker가 deterministic parse source입니다.

Memory

handoff + TODO

다음 루프 또는 다음 사람/agent가 같은 context에서 이어받습니다.

10. Safety Contract

금지 transport

one-shot advisor인 /ask, omc ask, omx ask, provider ask는 금지입니다.

/ask 금지omx ask 금지

허용 transport

omx/codex를 pane 안에 띄운 뒤 cmux send, set-buffer, paste-buffer로 prompt를 주입하는 방식은 persistent pane orchestration이므로 허용됩니다. omx exec, codex exec는 기본 금지입니다.

cmux sendread-screen

위험 flag 금지

--dangerously-skip-permissions, --yolo, -a never, --approval-mode never는 기본 사용 금지입니다.

Secret 보호

full env dump, secret file 읽기, token/key 출력은 금지입니다. 기록 전 secret-like env name과 literal key는 redaction 대상입니다.

Reviewer 권한 제한

reviewer는 기본적으로 source를 수정하지 않습니다. finding을 제시하고, orchestrator가 최종 ACCEPT/REJECT/NEEDS_CHECK/DEFER를 판단합니다.

11. 이 동작을 어떻게 검증하나

12. 전체 End-to-End 시나리오 예시

  1. 대상 프로젝트에서 사용자가 /cmux-agent-harness-loop loop "로그인 버그 고쳐줘"를 실행합니다.
  2. orchestrator는 setup이 없으면 scaffold하고, 요청을 bugfix로 분류합니다.
  3. TASK_SPEC.md에 원문 요청, 계획, acceptance criteria를 기록합니다.
  4. Claude/OMC orchestrator가 직접 source를 수정합니다.
  5. reviewer pane을 확보하고 9개 context를 prompt로 주입합니다.
  6. reviewer는 REVIEW_L001_1.md.tmp → REVIEW_L001_1.md → REVIEW_L001_1.md.done → sentinel 순서로 결과를 냅니다.
  7. orchestrator는 .done 확인 후 verdict를 파싱합니다.
  8. STOP이면 finding을 판정해 ACCEPT 항목은 TODO.md에 남기고 다음 루프로 들어갑니다.
  9. GO이고 남은 acceptance criteria가 없으면 DONE으로 수렴하고 HANDOFF.md를 남깁니다.
